#  Step 3: Security Hardening

## 2. Harden SSH authentication

This is the core of brute-force resistance.

Recommended SSH server settings:

* `PermitRootLogin no`
* `PasswordAuthentication no`
* `KbdInteractiveAuthentication no`
* `PubkeyAuthentication yes`
* `PermitEmptyPasswords no`
* `MaxAuthTries` set lower than default
* `AllowGroups sshadmins`
* `X11Forwarding no`
* `AllowAgentForwarding no`
* `AllowTcpForwarding no` unless you truly need it
* `PermitTunnel no`

These are all valid `sshd_config` controls documented in the OpenSSH server configuration manual. ([OpenBSD Manual Pages][2])

### Why this matters

* disabling password auth removes the most common SSH brute-force path
* disabling root login improves accountability and removes direct root-target login attempts
* restricting forwarding reduces abuse of the SSH daemon as a pivot/tunnel

Ubuntuâ€™s current SSH hardening guidance also recommends disabling root login, eliminating passwords, and restricting network access. ([Ubuntu][4])

## 3. Protect against brute force

Use multiple layers, not just one.

### Layer A: key-only SSH

With `PasswordAuthentication no`, attackers cannot brute-force account passwords through SSH in the usual way. `PasswordAuthentication` and `PubkeyAuthentication` are standard OpenSSH controls. ([OpenBSD Manual Pages][2])

### Layer B: Fail2ban

Install and enable Fail2ban for SSH.

Fail2ban is designed to detect authentication failures and ban offending IPs, and its jail configuration model is documented through `jail.conf`/`jail.local`. Ubuntuâ€™s package docs recommend customizations in `.local` files rather than editing shipped `.conf` files directly. ([Ubuntu Manpages][5])

Recommended SSH jail behavior in your script:

* enable SSH jail
* use a moderate `findtime`
* use a moderate `maxretry`
* use a meaningful `bantime`
* whitelist only trusted admin IPs if necessary

### Layer C: firewall source restriction

If the admin has a fixed office/VPN IP, restrict SSH to that source range instead of exposing it globally. Ubuntu guidance recommends firewalls and limiting access ranges as part of SSH hardening. ([Ubuntu Help][6])

## 4. Add first-line MITM protections for SSH administration

This is the part you explicitly asked for.

### Important truth

SSH already provides encrypted communication between untrusted hosts, but MITM protection depends on **correct host key verification**. OpenSSHâ€™s client docs state that `StrictHostKeyChecking yes` provides maximum protection against man-in-the-middle attacks, and changed host keys are refused. ([OpenBSD Manual Pages][7])

So Step 3 should include:

### Server-side actions

* generate/verify server host keys
* record and display host key fingerprints after setup
* save the fingerprints into your installer state file
* show the operator a post-step message telling them to verify fingerprints on first SSH login from client machines

### Client guidance your script should print

For admin machines, recommend:

* `StrictHostKeyChecking ask` or `yes`
* never ignore host key change warnings
* pre-load the server fingerprint into known_hosts where possible

OpenSSH also documents:

* `CheckHostIP yes` as a way to detect host key changes associated with DNS spoofing
* `VerifyHostKeyDNS yes|ask` for SSHFP-based verification
* `UpdateHostKeys` for graceful host-key rotation after trust is established. ([OpenBSD Manual Pages][1])

### Practical script behavior

At the end of Step 3, your script should print something like:

```text
SSH host fingerprints:
- ED25519: <fingerprint>
- ECDSA: <fingerprint>

Verify these on the admin workstation before accepting the first SSH connection.
Do not ignore host key change warnings.
```

That is the safest honest way to treat MITM here.

## 5. Lock down the network surface with a firewall

Use UFW with a deny-by-default inbound policy.

Ubuntuâ€™s UFW docs state that the typical secure baseline is:

* deny incoming
* allow outgoing
* explicitly allow only needed services. ([Ubuntu Help][6])

Recommended Step 3 firewall state:

* allow SSH only
* allow HTTP/HTTPS later when needed
* deny all other inbound by default
* optionally restrict SSH by source IP

At this stage:

* open SSH
* do **not** open database or Redis externally
* leave application ports closed until later steps

Since Step 5 will install many packages, Step 3 should ensure the host is prepared for safer package retrieval.

Actions:

* ensure system CA certificates are present
* keep package indexes updated from the distro package manager
* avoid unsigned third-party package sources unless explicitly approved
* store third-party repo keys in the modern signed-by style later, not legacy global trust

Iâ€™m keeping this high-level because the exact repo additions happen in Step 5, but this matters for supply-chain risk.

## 7. Reduce account and session abuse

Recommended SSH/session hardening:

* lower `MaxAuthTries`
* set reasonable idle disconnects with `ClientAliveInterval` and `ClientAliveCountMax`
* disable unused forwarding features
* disable environment injection features you do not need

These are all standard OpenSSH server options. ([OpenBSD Manual Pages][2])

This helps against:

* credential spraying
* unattended admin sessions
* use of SSH as a tunnel for unintended traffic
* some post-compromise abuse paths

## 9. Safe SSH change procedure

This is critical.

Your script should **not** disable root login or password auth until it has:

* confirmed a public key is installed for `BENCH_USER`
* validated `sshd_config`
* restarted/reloaded SSH safely
* warned the operator to test a second SSH session before closing the first one

That avoids lockout while hardening.
