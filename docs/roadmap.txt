
# Build roadmap

## Phase 1 — Bootstrap the repo

Goal: make the repo real and installable.

Build:

* create the exact folder structure
* add `pyproject.toml`
* add package entrypoints:

  * `src/frappectl/__init__.py`
  * `src/frappectl/__main__.py`
  * `src/frappectl/cli.py`
* add `README.md`, `LICENSE`, `.gitignore`
* add empty module files from your structure
* add `scripts/install.sh`
* add `scripts/dev-install.sh`

Done when:

* `pip install -e .` works
* `python -m frappectl` works
* `frappectl --help` works

---

## Phase 2 — Wire the CLI shell

Goal: make the CLI surface exist end-to-end.

Build:

* register all command groups in `cli.py`
* wire:

  * `setup`
  * `bench`
  * `site`
  * `apps`
  * `backup`
  * `update`
  * `jobs`
  * `production`
  * `ssl`
  * `diagnostics`
  * `menu`
* each command file gets at least one live placeholder command
* make global options work:

  * `--bench`
  * `--verbose`
  * `--yes`
  * `--non-interactive`

Done when:

* all top-level commands appear in help
* each command module is callable without import/runtime failure

---

## Phase 3 — Core foundation

Goal: build the shared engine everything else depends on.

Build:

* `core/constants.py`
* `core/errors.py`
* `core/paths.py`
* `core/logging.py`
* `core/registry.py`
* `core/context.py`
* `core/config.py`
* `core/state.py`

Core outcomes:

* bench registry file model exists
* active bench resolution exists
* per-bench config/state path resolution exists
* logging works consistently

Done when:

* code can register a bench
* code can resolve current bench
* code can load/save bench config
* code can load/save bench state

---

## Phase 4 — Execution/integration layer

Goal: safely run system commands and external tools.

Build:

* `integrations/shell.py`
* `integrations/bench.py`
* `integrations/systemd.py`
* `integrations/apt.py`
* `integrations/git.py`
* `integrations/ssh.py`
* `integrations/mariadb.py`
* `integrations/redis.py`
* `integrations/nginx.py`
* `integrations/supervisor.py`

Important:

* centralize subprocess execution
* support:

  * normal execution
  * sudo/root execution
  * run-as-bench-user execution
  * dry-run later if needed
* standardize return values and errors

Done when:

* one shell runner is used everywhere
* no command module directly shells out on its own

## Phase 6 — Setup engine scaffold

Goal: make the 12-step setup workflow executable.

Build:

* `setup/models.py`
* `setup/planner.py`
* `setup/engine.py`
* `setup/steps/__init__.py`
* register all 12 steps
* implement:

  * `setup run`
  * `setup step`
  * `setup status`
  * `setup resume`

Behavior:

* each step has a clear interface
* step progress is saved in bench state
* failed steps stop cleanly
* resume continues from saved state

Done when:

* setup engine can enumerate and run steps
* step status persists across runs

## Phase 13 — Installer actualization

Goal: make the repo install the CLI from Git and make it usable immediately.

Build:

* finalize `scripts/install.sh`
* finalize `scripts/dev-install.sh`

Installer responsibilities:

* verify OS assumptions
* ensure Python/pip exists
* install package from Git or local clone
* verify `frappectl` command is available
* print next step:

  * `frappectl setup run`
  * or `frappectl setup step 1`

Done when:

* the repo can be installed from Git
* the CLI becomes available immediately after install
